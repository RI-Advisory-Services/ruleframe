## Focus Areas (in priority order)

### 1. Bundle Loading API Redesign â€” DONE

**Problem:** `RuleBundle` is a thin wrapper around `raw: dict`. It only accepts file paths for YAML. The design doc says consumers will load rules from S3, database fields, API responses, etc. â€” but there's no way to pass content directly. Additionally, structural problems (missing `id`, bad `fail_when`, unknown types) aren't caught until deep inside `validate_dataframe`.

**Goal:** Make `RuleBundle` the validated entry point that accepts content from any source and fails fast with clear errors.

**What was done:**
- Added `RuleBundle.from_yaml_string(content: str)` â€” accepts raw YAML text from any source (S3, DB, API)
- Added `RuleBundle.from_json_string(content: str)` â€” accepts raw JSON text
- Added `__post_init__` with `_validate_structure()` for eager validation at construction
- Refactored `from_yaml(path)` to delegate to `from_yaml_string` (DRY)
- Structural validation checks: `rules` is a list, each rule is a dict with `id` and `fail_when` (dict), `computed_columns` is a list
- Backward compatible: `from_yaml(path)` and `from_json_dict(dict)` still work identically
- Updated 2 existing tests that expected lazy validation on property access (now fires at construction)
- Added 9 new tests: `from_yaml_string`, `from_json_string`, invalid JSON, missing `id`, missing `fail_when`, non-dict `fail_when`, non-dict rule entry

**Files changed:** `src/ruleframe/bundle.py`, `tests/test_bundle.py`

### 2. NullSafeComparison Operator Pattern â€” DONE

**Problem:** The `NullSafeComparison` class uses `cls._op` as a class-level static method, then passes it as a `_func` dataclass field via `from_expression`. This:
- Confuses mypy (`"type[NullSafeComparison]" has no attribute "_op"`)
- Is non-obvious to contributors
- Fights Python's data model (dataclass fields + class-level callables)

**What was done:**
- Removed `_func: Any` dataclass field entirely
- Removed all `_op = staticmethod(lambda ...)` class variables
- Added `_compare(self, left: Any, right: Any) -> bool` method on `NullSafeComparison` (raises `NotImplementedError`)
- Each subclass (`NullSafeEq`, `NullSafeGt`, etc.) implements `_compare` as a one-liner
- `NullSafeNotEq` keeps its custom `evaluate` override for "blank left means fire" semantic, calls `self._compare`
- `from_expression` no longer passes `_func=cls._op` â€” just constructs with `left` and `right`
- Eliminated the mypy `attr-defined` error (project-wide mypy errors: 11 â†’ 10)

**Files changed:** `src/ruleframe/operators.py`

### 3. Vectorized Date Computed Columns â€” DONE

**Problem:** `_compute_date_diff`, `_compute_days_since_today`, `_compute_years_since_year`, and `_compute_all_blank_or_zero` use `df.apply(axis=1)` which is row-by-row Python iteration. For 100k+ row workbooks this becomes a bottleneck.

**What was done:**
- Added `_to_datetime_series()` helper: tries `pd.to_datetime(errors='coerce')` first, then patches only failed cells with `dateutil.parse` fallback. This preserves correctness for messy mixed-format Excel data while being ~100x faster on clean columns.
- `_compute_date_diff`: vectorized `(end_dt - start_dt).dt.days` with `Float64` nullable dtype
- `_compute_days_since_today`: vectorized `(reference - col_dt).dt.days`
- `_compute_years_since_year`: vectorized `pd.to_numeric(errors='coerce')` then arithmetic
- `_compute_all_blank_or_zero`: vectorized per-column `isna | is_zero | is_empty_str` combined with `&`
- Updated 1 test assertion from `is None` to `is None or pd.isna()` (nullable Float64 returns `pd.NA` not `None`)
- Verified hybrid dateutil fallback works with mixed date formats ("Jan 1, 2024", "2024-03-01", "03/15/2024")

**Files changed:** `src/ruleframe/computed.py`, `tests/test_computed.py`
