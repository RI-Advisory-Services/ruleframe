# Code Audit — 2026-05-30

## Focus Areas (in priority order)

### 1. Bundle Loading API Redesign

**Problem:** `RuleBundle` is a thin wrapper around `raw: dict`. It only accepts file paths for YAML. The design doc says consumers will load rules from S3, database fields, API responses, etc. — but there's no way to pass content directly. Additionally, structural problems (missing `id`, bad `fail_when`, unknown types) aren't caught until deep inside `validate_dataframe`.

**Goal:** Make `RuleBundle` the validated entry point that accepts content from any source and fails fast with clear errors.

**Changes:**
- Add `from_yaml_string(content: str)` classmethod
- Add `from_json_string(content: str)` classmethod  
- Add eager structural validation at construction time (version present, rules have `id` + `fail_when`, computed columns have valid `type`)
- Keep `raw` for introspection but make construction the validation boundary
- Preserve backward compat: `from_yaml(path)` and `from_json_dict(dict)` still work

**Design constraint:** Don't over-validate. Check structure (required keys, correct types), not semantics (whether referenced columns exist — that still belongs at evaluation time since it depends on the input DataFrame).

### 2. NullSafeComparison Operator Pattern

**Problem:** The `NullSafeComparison` class uses `cls._op` as a class-level static method, then passes it as a `_func` dataclass field via `from_expression`. This:
- Confuses mypy (`"type[NullSafeComparison]" has no attribute "_op"`)
- Is non-obvious to contributors
- Fights Python's data model (dataclass fields + class-level callables)

**Goal:** Same behavior, idiomatic pattern that mypy and humans understand.

**Approach:** Replace `_func` dataclass field with an abstract method `_compare(self, left, right) -> bool` that subclasses override. The `evaluate` method calls `_compare` after null checks. `NullSafeNotEq` keeps its custom `evaluate` override for the "blank left means fire" semantic.

### 3. Vectorized Date Computed Columns

**Problem:** `_compute_date_diff`, `_compute_days_since_today`, and `_compute_all_blank_or_zero` use `df.apply(axis=1)` which is row-by-row Python iteration. For 100k+ row workbooks this becomes a bottleneck.

**Goal:** Vectorized implementations using pandas datetime operations where possible.

**Approach:** 
- Use `pd.to_datetime(series, errors='coerce')` for bulk parsing
- Fall back to per-row `dateutil.parse` only for cells that `pd.to_datetime` couldn't handle
- `date_diff` becomes: `(pd.to_datetime(end) - pd.to_datetime(start)).dt.days`
- `days_since_today` becomes: `(today - pd.to_datetime(col)).dt.days`
- `all_blank_or_zero` becomes: vectorized null check + numeric coercion

**Risk:** `pd.to_datetime` is stricter than `dateutil.parse` on mixed formats. The hybrid approach (vectorize first, patch failures with dateutil) preserves correctness while getting 95%+ of the speed benefit.
